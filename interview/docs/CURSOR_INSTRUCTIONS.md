## 0. 最初に必ず読むもの（読み込み順）

2. 本書のセクション2（絶対制約）

## 1. プロジェクト概要（1分版）

中星工業の人事評価マニュアルに準拠した、AIロールプレイ型1on1面談研修アプリ。
リアルタイム音声対話が主体機能。シーン2種（期初目標設定/期末評価面談）×役割2種（上司役/部下役）×難易度3段階（中小企業/大企業/コンサルタント）。
面談終了後に審判AIが「良い点/改善すべき点/全体的な評価/総評」を出力する。

## 2. 絶対制約（違反禁止）

- **C-01** 動作環境は Ubuntu / `/opt/interview` / venv。パスをハードコードせず `Path(__file__)` 基準または環境変数で解決する。
- **C-02** 純Python＋外部API呼び出しのみ。ローカルLLM推論、CPUアーキテクチャ依存のネイティブバイナリ、コンパイル必須の特殊依存を導入しない（サーバー更改予定のため）。
- **C-03** `.env` は `/opt/interview/.env`。作成時にパーミッション600を設定し、`.gitignore` に登録する。APIキーをログ・例外メッセージ・フロントエンドに一切出力しない。
- **C-04** 等級マスタは7等級（1-1/1-2/1-3, 2, 3, 4-1, 4-2, 5, 6, 7）。8等級・取締役を追加しない。
- **C-05** 利用者情報に性別を含めない（入力欄・DB列・プロンプトのいずれにも）。
- **C-06** 音声対話の基準実装はGemini Live API。他プロバイダ（OpenAI/Anthropic/xAI）は審判・ペルソナ生成等のテキスト役割のみ（初版）。
- **C-07** UIフレームワークはFastAPI＋静的フロントエンド（素のHTML/JS）。Streamlit/Gradioを使わない（リアルタイム音声との相性のため）。
- **C-08** 音声データ自体はDBに保存しない。トランスクリプトのみ保存。
- **C-09** モデル名・Live APIのプロトコル仕様は変化が速い。**実装前に必ず公式ドキュメントの最新情報を確認**し、モデル名は`.env`から読み込む設計にする（コードに直書きしない）。
- **C-10** 判定ロジック（禁止表現・評語基準等）をプロンプト内に直書きせず、`data/rubrics.json` から読み込んで組み立てる。マニュアル改定時にJSONの差し替えだけで済むようにする。

## 3. 技術スタック

## 5. 実装フェーズとタスク

各フェーズ完了時にテストを通し、変更概要を `docs/CHANGELOG.md` に追記してから次へ進むこと。

### P1: 基盤
- [ ] venv構築手順・requirements.txt・README（セットアップ、mkcertによるHTTPS手順、systemdサービス例）
- [ ] `config.py`: .env読込、役割別プロバイダ解決（ROLE_PARTNER/ROLE_JUDGE/ROLE_PERSONA、未指定はgemini）
- [ ] `providers/`: 共通インターフェース `generate(system, messages, json_mode: bool) -> str` を持つ gemini/openai/anthropic/xai アダプタ
- [ ] `db.py`: users/sessions/turns/reports テーブル（SPECIFICATION.md セクション8）
- [ ] `data/grades.json` / `data/rubrics.json` 作成（SPECIFICATION.md 3.2/3.9の全項目を構造化。禁止表現リスト、器具高そぅ、目標値ルール、状態目標事例10種、評語基準、面談5ステップ、ほめ方8スキル、評価者エラー6類型、ギャップ4パターン、等級別方式・ウェイトを漏れなく含める）
- [ ] `checker.py`: 禁止表現regex検出（行動目標表現20語＋以上/以下/最大/最小）
- **受入基準**: pytest緑。CLIスクリプトでテキスト1ターン対話とチェッカーが動作。

### P2: テキストモード完成
- [ ] `persona.py`: 利用者プロフィール＋モード＋難易度からペルソナJSON生成（性格特性、hidden_facts、初期申告水準%）。上司役ペルソナはマニュアルの評価者区分（主任の一次評価者=係長 等）に整合させる
- [ ] 相手役対話サービス: 4モード別システムプロンプト。口調指定を含む（AI上司=丁寧だが敬語に拘らず砕けた口調 / AI部下=敬語意識＋少しダルい口調・一人称や語尾で男性性を表現）
- [ ] `judge.py`: 役割別ルーブリックでJSON採点（観点別1-7点、良い点/改善点/全体評価/総評/模範例）。難易度によるリテラシー水準差・詳細度差（詳細度差は小さく）をプロンプトで制御。「5=期待水準通り、安易に高得点をつけない」を審判プロンプトに明記
- [ ] 終了判定: 利用者宣言 / AI宣言（1B・2BでAI上司がゴール到達を判断） / 時間・ターン上限
- [ ] 画面 S-01〜S-05（テキスト対話で動作）
- **受入基準**: 4モード×3難易度がテキストで一連動作（設定→面談→講評→履歴保存）。禁止表現を含む発言に画面警告が出る。1AでAI部下が85-95水準到達時に審判が合格判定を出す。

### P3: 音声モード（主体機能）
- [ ] `live_bridge.py`: ブラウザ⇔サーバー⇔Gemini Live のWebSocketプロキシ。音声フォーマット変換（ブラウザPCM⇔Live API要求形式）、バージイン、トランスクリプト受信
- [ ] フロント音声UI: マイク取得、ストリーミング再生、音声⇔テキストのリアルタイムトグル（文脈維持）、発話中インジケータ
- [ ] ボイス割当: 部下役=男性系既成ボイス、上司役=中性〜落ち着いた既成ボイス。最新のボイス一覧をドキュメントで確認し `.env` で変更可能にする
- [ ] 音声ターンもturnsテーブルにトランスクリプトとして保存
- **受入基準**: HTTPS環境（mkcert）で音声面談→途中でテキスト切替→音声復帰→講評まで一連動作。AI発話への割り込みが機能する。

### P4: 運用機能
- [ ] S-06 管理者画面（ADMIN_PASSWORD認証、全履歴、部門別/観点別集計）
- [ ] マイページのスコア推移グラフ
- [ ] 同時音声セッション数の上限制御（.env設定）
- [ ] S-01の同意表示（ログ保存・総務閲覧・外部API送信・実在情報入力禁止）
- **受入基準**: 複数ブラウザからの同時利用で干渉しない。管理者が全履歴を閲覧できる。

## 6. プロンプト設計の要点

- 相手役AIと審判AIを分離する（演技と採点を同一コンテキストでさせない）。
- 相手役プロンプトには「メタ発言禁止」「1発話2〜5文」「利用者のスキルに応じた反応変化（傾聴されれば開示、高圧的なら萎縮/反発、二者択一質問には短答のみ）」を含める。
- hidden_factsは相手役プロンプト内にのみ置き、フロントエンドに送らない。
- 審判の出力はJSONスキーマを明示し `json_mode` で受ける。パース失敗時は1回リトライ。
- ルーブリック内容はrubrics.jsonから組み立てる（C-10）。

## 7. コーディング規約

- 型ヒント必須。ruffでlint（設定はpyproject.tomlに）。
- 例外はアプリ層でハンドリングし、利用者向けには日本語の平易なエラー表示。API障害時は「再試行」導線を出す。
- LLM呼び出しは全てproviders経由。直接SDKを叩くのはproviders内のみ。
- ログ: 標準loggingで `logs/app.log`（回転）。発話内容はDEBUGレベルでのみ、APIキーは絶対に出さない。
- UI文言はすべて日本語。

## 8. やってはいけないこと

- 仕様にない機能の先行実装（提案は `docs/PROPOSALS.md` に書き、人間の承認を待つ）
- rubrics.jsonの内容の改変（マニュアル準拠のため。誤記を見つけたら報告）
- 依存パッケージの無断大量追加（1フェーズあたり新規追加は要点のみ、READMEに理由を記載）
- テスト省略でのフェーズ進行
- `.env` やDBファイルのコミット
