## 1. C-09事前調査（実装前必須・受入条件）

Gemini Live API は変更が速いため、**学習済み知識や過去のサンプルコードを根拠に実装してはならない**。
実装着手前に、Google公式ドキュメント（ai.google.dev / cloud.google.com の当該ページ）を参照し、
以下を確認して docs/notes/live_api_survey.md に出典URL・確認日付きで記録すること。

- [ ] Live API対応の**最新モデル名**（音声対話用のネイティブオーディオ系モデル。preview/GA区分と提供状況を含む）
- [ ] **接続プロトコル**（WebSocketエンドポイント、セッション確立手順、setup/config メッセージの現行スキーマ）
- [ ] **認証方式**（サーバー側APIキー保持を前提とした場合の、ブラウザ⇔FastAPI⇔Google間の中継構成。
      エフェメラルトークンが現行提供されているなら、その利用可否と有効期限）
- [ ] 音声フォーマット要件（入力/出力のサンプルレート・エンコーディング）
- [ ] **バージイン（割り込み発話）**の公式サポート状況と、割り込み時のサーバーイベント仕様
- [ ] セッション時間・同時接続・レート制限（15分面談1回を1セッションで完走できるか、再接続戦略が必要か）
- [ ] 料金体系（音声入出力のトークン/秒課金の別）

調査の結果、仕様書記載のモデル名やプロトコルが現行と異なる場合は、
**実装せず差分を報告し、仕様書更新の指示を待つこと。**

## 2. C-09 実装スコープ

- FastAPI側: ブラウザとGoogle Live APIを中継するWebSocketプロキシ
  （APIキーをブラウザへ渡さないこと。1章でエフェメラルトークン方式が妥当と確認できた場合はその方式でも可。採用理由をnotesに記録）
- フロント側: マイク取得（getUserMedia）、音声再生、テキストモードと共通のセッション/ペルソナ/審判パイプラインへの合流
- 面談ログ: 音声セッションでも発話をテキスト化してDBに保存し、**審判はテキストモードと同一のjudge.pyを通す**こと
  （音声用に別審判を作らない。寛大化ドリフト防止のため）
- 禁止表現検知: 音声書き起こしに対しても現行と同じadvisory表示とする（P2.1-3の修正後仕様に従う）

## 3. HTTPS環境（受入条件）

getUserMediaはsecure context必須のため、LAN内検証はHTTPSで行う。

- [ ] mkcertでローカルCAを作成し、サーバーのLAN内ホスト名/IPに対する証明書を発行
- [ ] uvicornにcert/keyを設定しHTTPSで起動（起動手順をREADMEに追記）
- [ ] 検証用クライアントPCにmkcertのルートCAを導入し、警告なしでマイク許可ダイアログが出ることを確認
- [ ] 証明書ファイルは .gitignore に追加

## 4. バージイン動作確認（受入条件）

- [ ] AIの音声応答の途中で利用者が発話した際、AI音声の再生が停止（または即時減衰）し、利用者発話が受理される
- [ ] 割り込み後のAI応答が、割り込み前の応答の続きではなく利用者発話を踏まえた内容になる
- [ ] 割り込みで打ち切られたAI発話が、面談ログ上で「実際に再生された範囲」と区別可能な形で記録される
      （審判が「言っていないこと」を評価材料にしないため）
- [ ] 上記をS1相当の設定（1A/中小企業）で1セッション通しで実施し、docs/acceptance/P3/ に証跡保存

## 5. P3受入条件まとめ

1. docs/notes/live_api_survey.md が存在し、最新モデル名・プロトコルの出典URL・確認日が記録されている（1章）
2. mkcertによるHTTPS環境でマイク取得〜音声対話が動作する（3章）
3. バージインが4章の4項目すべてを満たす
4. 音声セッションの審判出力がテキストモードと同一スキーマ・同一judge.pyで生成される
5. pytest既存27件がグリーンを維持し、音声経路のテストが追加されている
