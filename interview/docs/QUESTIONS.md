## Q-01: P1 受入基準の解釈差（SPEC vs CURSOR_INSTRUCTIONS）

| 文書 | P1 完了条件 |
|---|---|
| `SPECIFICATION.md` セクション9 | 「テキストのみで **1A/1B の対話**が CLI または最小 UI で成立」 |
| `CURSOR_INSTRUCTIONS.md` セクション5 P1 | 「CLI スクリプトでテキスト **1ターン対話**とチェッカーが動作」 |

**論点**: P1 で 1A/1B の「対話」とは、1ターンの往復で足りるか、それともモード別システムプロンプトを使った複数ターンの簡易セッション（ペルソナ生成・終了判定・講評なし）まで必要か。

**選択肢**:
- **A**: 1ターン往復＋チェッカー動作で十分（基盤検証）。1A/1B の本格対話は P2。
- **B**: モード 1A/1B それぞれで最低 3〜5 ターンの簡易 CLI セッションが動くこと（`persona.py` の一部を P1 に前倒し）。
- **C**: その他（具体的に指定）

**推奨案**: **A**。`persona.py`・審判・4モード完成は P2 タスクとして明記されており、P1 は providers/DB/rubrics/checker の基盤に集中するのが自然。ただし SPEC の文言との整合のため、P1 完了時点では「1A/1B 用のモード識別子とプロンプト骨格を CLI から指定できる」程度を最低ラインとする。

**回答**: A を採用。P1は1ターン往復＋チェッカー動作まで。ただしあなたの提案どおり、CLIから1A/1Bのモード識別子とプロンプト骨格を指定できる状態を最低ラインとする。
