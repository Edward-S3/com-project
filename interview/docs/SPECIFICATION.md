## 1. 目的と背景

### 1.1 目的

人事評価制度がマニュアルどおりに運用されていない主因は、評価者・被評価者双方の面談スキルと目標設定スキルの不足にある。本アプリは、AIとの1on1面談ロールプレイを通じて次の2点を習熟させる社内研修ツールである。

1. **評価者の立場**: 部下への適切な指導（面談5ステップ、傾聴・質問・ほめ方、事実に基づく評価、評価者エラーの回避）ができるようになる。
2. **被評価者の立場**: 自身の職位・技術・勤続年数に応じた適切な目標設定（SMART＝器具高そぅ、禁止表現の排除、評価基準の設定）と、評価面談での合理的な自己申告ができるようになる。

### 1.2 特に想定する課題（原案より）

- 昇進・昇格して部下を持った社員が、必ずしも適切な指導を行えるとは限らない。
- 地方の中小企業では、監督職・管理職による部下評価が正しく行われないことが多い。
- 社内アンケートでも「評価基準が不明確」62.8%等の不満が確認されている。

## 2. 全体像

### 2.1 シミュレーションの構成軸

利用者はセッション開始時に以下を選択する。

| 軸 | 選択肢 |
|---|---|
| シーン | (1) 期初の目標設定面談 / (2) 期末の評価面談 |
| 役割 | (A) 上司役（AIが部下役） / (B) 部下役（AIが上司役） |
| 難易度 | 中小企業 / 大企業 / コンサルタント |
| 入出力方式 | 音声対話（主体） / テキスト対話（検証用） |

シーン×役割の4通り（1A, 1B, 2A, 2B）をすべて実装する。

### 2.2 4モードの概要

| モード | 利用者 | AI | 練習内容 | 合格イメージ |
|---|---|---|---|---|
| 1A 期初×上司役 | 上司 | 部下 | 部下の不適切な目標申告（低水準・曖昧）を対話で適正水準に誘導する | AI部下が適正難易度（後述の85〜95水準）かつSMARTを満たす目標を自ら申告するに至る |
| 1B 期初×部下役 | 部下 | 上司 | 自分の等級・職務に適した目標を申告し、上司の指摘に合理的に応答・修正する | 禁止表現なし・数値または状態基準つきの目標で上司の承認を得る。変更申告は理由＋変更後内容をセットで行う |
| 2A 期末×上司役 | 上司 | 部下 | 実績と自己評価にギャップのある部下への評価フィードバック（ギャップ4パターン対応） | 事実確認→ねぎらい→期待水準の提示→次期への動機づけの流れを実践できる |
| 2B 期末×部下役 | 部下 | 上司 | 目標に対する実績と自己評価を申告し、上司評価との齟齬を事実ベースで検証される | 自己評価の根拠を事実・数値で説明でき、上司評価との差の理由を理解した状態で面談を終える |

## 3. 機能要件

### 3.1 利用者情報の入力（F-01）

セッション開始前に以下を入力する。**性別は取得しない**（評価・指導の適切さと無関係であり、バイアス混入を避けるため）。

| 項目 | 形式 | 用途 |
|---|---|---|
| 氏名 | テキスト | 履歴管理、面談中の呼びかけ |
| 所属部門 | 選択式＋自由入力 | ペルソナと目標テーマの文脈生成 |
| 年齢 | 数値 | ペルソナ調整 |
| 勤続年数 | 数値 | 期待水準の調整 |
| 職位（等級） | 選択式（3.2の等級マスタ） | 成果評価方式・ウェイト・期待水準の決定 |

2回目以降はプロフィールを保存済みリストから選択できる（複数利用者対応）。

### 3.2 等級マスタ（F-02）

中星工業の資格等級制度（7等級）に完全準拠する。**取締役（8等級）は設けない**（現行制度の評価対象外のため）。

| 等級 | 対応役職 | 適用される成果評価方式 |
|---|---|---|
| 1級（1-1高卒 / 1-2短大専門卒 / 1-3大卒） | 一般 | QCD項目＋事後申告 |
| 2級 | 一般 | QCD項目＋事後申告 |
| 3級 | 主任 | QCD項目＋事後申告 |
| 4-1級 | 係長 | 方針管理との連動＋事後申告 |
| 4-2級 | 課長代理 | 方針管理との連動 |
| 5級 | 課長 | 方針管理との連動 |
| 6級 | 次長・部長代理 | 方針管理との連動 |
| 7級 | 部長 | 方針管理との連動 |

昇給・昇格ウェイト（成果/プロセス）: 1〜2級 30/70、3級 40/60、4-1級 50/50、4-2級以上 60/40。賞与ウェイト: 1〜2級 40/60、3級 50/50、4-1級 60/40、4-2級以上 70/30。講評時の説明に使用する。

### 3.3 音声対話（F-03）【本アプリの主体機能】

- リアルタイム双方向音声対話を初版から実装する。基準実装はGemini Live API（ネイティブ音声対話モデル）とする。
- ブラウザのマイクから音声入力し、AIの音声応答をストリーミング再生する。割り込み（バージイン: AI発話中に利用者が話し始めたらAIが発話を止めて聴く）に対応する。
- **口調指定**（システムプロンプトによる話し方制御＋ボイス選択で実現する）:
  - 利用者が**部下役**（AI＝上司役）の場合: 丁寧な口調でありながら、敬語の使用に拘らない砕けた話し方。威圧的にはしない。
  - 利用者が**上司役**（AI＝部下役）の場合: **男性の声**。敬語を意識しているが、少しダルさ（気だるさ・覇気のなさ）を感じる口調。
  - ボイスは各APIの既成ボイスから性別・声質の近いものを選択し、口調のニュアンスはプロンプトで制御する（実装時に最新のボイス一覧を確認して割り当てる）。
- 画面には常時トランスクリプト（発話の文字起こし）を表示する。

### 3.4 テキスト対話とリアルタイム切替（F-04）

- テキスト対話モードは検証環境（音声を出せない場所）向けの位置づけとする。機能・判定は音声モードと同一。
- セッション実行中の切替: 音声セッション中もテキスト入力欄を常時有効とし、テキストで入力すればAIはテキスト（＋任意で音声）で応答する。音声⇔テキストをUI上のトグルでいつでも切替でき、会話の文脈は維持される。

### 3.5 AIペルソナ生成（F-05）

- AIの相手役は、利用者が入力したプロフィール（部門・年齢・勤続年数・等級）に基づいて動的に生成する。
  - AI部下役: 利用者の部下として自然な等級・年齢構成（原則、利用者より1〜2等級下）。
  - AI上司役: 利用者の直属上司として自然な役職（等級マスタの評価者区分に準拠。例: 主任の一次評価者は係長）。
- ペルソナには性格特性（自己評価過大 / 謙虚すぎ / 方針非連動・曖昧表現多用 など）を割り当てる。特性は難易度と組み合わせて選択される（3.6参照）。
- AI部下役は「隠し情報（hidden facts）」を持ち、利用者が傾聴・具体的質問をしない限り重要な事実を開示しない。「事実に基づく評価」の必要性を体感させるための設計である。

### 3.6 難易度3段階（F-06）

難易度は「AI側の世間一般的な知識・教養に対するリテラシーの高さの違い」と定義し、判定の厳しさ・講評の深さに反映する。詳細度の差は小さくする（「中小企業」でも「大企業」にやや劣る程度）。

| 難易度 | AIのリテラシー水準（判定・対話の深さ） | 講評の詳細度 |
|---|---|---|
| 中小企業 | 中星マニュアルの基本要件で判定（禁止表現なし、数値または状態基準あり、期限あり）。一般常識レベルの指摘 | 標準（大企業比でやや簡素） |
| 大企業 | 器具高そぅ5観点の完全充足、目標値5以下への3段階評価基準設定、部門方針との整合まで要求。人事・労務の一般教養を踏まえた指摘 | 標準 |
| コンサルタント | 上記に加え「評価点7＝真の目的」の水準を問う。目標の戦略的整合性、費用対効果、組織への波及効果まで対話で詰める。経営・人事理論の知見を引用した指摘 | 詳細 |

**AI部下の初期申告水準（モード1A）**: 「本人の技術・職務経験に対する最適目標＝100」としたとき、AI部下は難易度とペルソナ特性の組み合わせで決まる低水準の目標を申告してくる。利用者の誘導により**85〜95の水準**に到達すれば合格とする。

| | 中小企業 | 大企業 | コンサルタント |
|---|---|---|---|
| 控えめペルソナ | 80%水準を申告 | 70%水準 | 60%水準 |
| 過大自己評価ペルソナ | 80%水準（ただし本人はS級と主張） | 70%水準 | 60%水準 |

（水準の妥当性判定と誘導達成の判定は審判AIが行う。数値は内部パラメータとして保持し、講評で開示する。）

### 3.7 面談の進行と終了（F-07）

- 面談5ステップ（導入→趣旨説明→ふり返り→目標設定→クロージング）を画面上にガイド表示する。
- 面談の終了は次のいずれかで成立する。
  - 上司役（利用者またはAI）が終了を宣言する。
  - 部下役が利用者の場合、AI上司が「面談のゴールに達した／これ以上の進展が見込めない」と判断して終了を宣言する。
  - 利用者が終了ボタンを押す（どのモードでも可）。
- セッションには上限時間（既定30分、設定可）とターン数上限を設け、超過時はAIがクロージングに誘導する。

### 3.8 講評レポート（F-08）

面談終了後、審判AIがトランスクリプト全体を判定し、以下を出力する。

1. **良い点**（発言を引用して具体的に）
2. **改善すべき点**（発言を引用し、マニュアルのどの原則に反するかを明記）
3. **全体的な評価**（観点別スコア。7段階、5＝マニュアルの期待水準通り。レーダーチャート表示）
4. **総評**（次回への具体的アドバイス、模範解答例1件を含む）

観点は役割別に設定する。

- 上司役: 導入・緊張緩和 / 趣旨説明 / 傾聴・質問 / 事実に基づく確認 / ほめ方・承認 / 目標設定指導（1A）または評価フィードバック（2A） / クロージング・動機づけ、および評価者エラー6類型（寛大化・ハロー効果・中心化・対比誤差・イメージ評価・逆算評価）の兆候検出
- 部下役: 期限設定(き) / 具体性(ぐ) / 達成可能性(た) / 関連性(か) / 測定可能性(そ) / 評価基準の設定力 / 主体性・確認力（1B）、自己評価の根拠提示力・齟齬の受容と対話力（2B）、および禁止表現の使用検出

審判AI自体にも評語原則「5＝期待水準通り。安易に高評価をつけない（寛大化の回避）」を適用する。

### 3.9 判定ルーブリック（F-09）

中星工業マニュアルから構造化した以下のルールをJSONデータとして保持し、審判AIのプロンプトおよび即時チェッカーが参照する。

- 評語基準: S=7 / A=6 / B=5 / C=4 / D=3（成果・プロセス両基準の定義文を含む）
- 目標設定禁止表現: 行動目標表現（迅速化、効率化、向上、実施、算出、企画、推進、定着、図る、管理、共有化、育成、支援、徹底、強化、把握、目指す、浸透、検討、遂行）、不明確基準（以上、以下、最大、最小）、イチゼロ判定の回避
- 良い例: 件数、回数、日数、工数、率、精度、歩留、金額、原価、費、手番
- 器具高そぅ（SMART）5観点と良い例・悪い例
- 目標値設定ルール: 6以上=達成率換算 / 5以下=3段階評価基準必須 / 1=状態評価値 / 100%=状態評価値またはKPI100 / 定性目標=状態基準必須
- 状態目標事例集（作業実施、システム導入、基準作成、制度改訂、データ集計、活動推進、業務引継ぎ、情報収集、育成指導、納期遵守の各3〜7点スケール）
- 原則: 目標値＝上長の期待値＝評価点5。評価点7はその先（真の目的）を考える
- 面談5ステップの実施ポイント、ほめ方8スキル、承認4パターン、質問6パターン
- 評価者エラー6類型と対策
- 上司・部下評価ギャップ4パターン別対応
- 等級別成果評価方式・処遇反映ウェイト

テキスト・音声トランスクリプト双方に対し、禁止表現は正規表現による即時検出（画面警告）とLLMによる文脈判定の二重チェックとする。

### 3.10 履歴管理と管理者閲覧（F-10）

- 全セッション（プロフィール、モード、難易度、トランスクリプト、講評、スコア）をSQLiteに保存する。
- 利用者は自分の履歴と観点別スコアの推移（成長ログ）を閲覧できる。
- **管理者（総務等）は全利用者の履歴を閲覧できる**。管理者画面は`.env`に設定したパスワードによる簡易認証とする（実装時判断事項#1）。
- 利用開始時に「面談ログが保存され総務等が閲覧できること」「内容が外部AI APIに送信されること」「実在の人事情報・機密情報を入力しないこと」を画面で明示し、同意を得る。

## 4. LLM構成（F-11）

### 4.1 基準構成

Gemini APIを基準とする。音声対話はGemini Live API（ネイティブ音声）、審判・ペルソナ生成はGemini通常API。

### 4.2 マルチプロバイダ対応

`.env`にAPIキーが記載されている場合、OpenAI / Anthropic / xAI を利用可能とする。実装方式は以下の2つを両立する。

- **(a) 役割分担オーケストレーション**: 機能ごとに使用プロバイダを`.env`で指定できる（例: 相手役=Gemini Live、審判=Anthropic、ペルソナ生成=OpenAI）。未指定の役割はGeminiにフォールバック。
- **(c) UI選択**: 設定画面で、キーが登録済みのプロバイダの中から役割ごとの使用モデルを切替できる。UI設定は(a)のデフォルトより優先する。

音声対話（リアルタイム）はプロバイダごとに方式が大きく異なるため、初版の音声はGemini Live APIのみ対応とし、他プロバイダはテキスト系の役割（審判・ペルソナ生成・要約）で使用する（実装時判断事項#2）。

### 4.3 .env 定義（例）

```
GEMINI_API_KEY=...
OPENAI_API_KEY=...          # 任意
ANTHROPIC_API_KEY=...       # 任意
XAI_API_KEY=...             # 任意
ROLE_PARTNER=gemini          # 相手役（音声）: 初版はgemini固定
ROLE_JUDGE=gemini            # 審判: gemini/openai/anthropic/xai
ROLE_PERSONA=gemini          # ペルソナ生成
MODEL_GEMINI_LIVE=<実装時に最新のLive APIモデル名を確認して設定>
MODEL_GEMINI_TEXT=gemini-2.5-flash
ADMIN_PASSWORD=<管理者画面用>
SESSION_MAX_MINUTES=30
```

## 5. 非機能要件

| ID | 要件 |
|---|---|
| N-01 | 動作環境: Ubuntu Linux。`/opt/interview` 配下のPython仮想環境（venv）で動作する。`.env`も同フォルダに設置する |
| N-02 | CPUアーキテクチャ非依存: 純Python＋外部API呼び出しのみで構成し、ローカルLLM推論・ネイティブ拡張依存を持たない（後継サーバーハードウェア未定のため） |
| N-03 | セキュリティ: `.env`はパーミッション600、`.gitignore`登録。APIキーをログ・画面に出力しない |
| N-04 | **HTTPS必須**: ブラウザのマイク入力（getUserMedia）はセキュアコンテキストでのみ動作する。LAN配信時は自己署名証明書またはmkcert等でHTTPSを構成する（localhostでの検証時は不要） |
| N-05 | 多人数利用: 複数利用者の同時セッションに対応する。Live APIの同時接続数・レート制限を考慮し、同時音声セッション数の上限を設定可能とする |
| N-06 | ログ保護: SQLiteファイルはOSレベルのアクセス権で保護する。DBには利用者の発話がそのまま保存されることを前提に運用する |
| N-07 | 監査可能性: 全セッションにタイムスタンプ・使用モデル・難易度・モードを記録する |

## 6. アーキテクチャ

### 6.1 構成

```
[ブラウザ]  マイク/スピーカー + トランスクリプトUI (HTML/JS, Web Audio API)
    |  WebSocket (音声ストリーム・イベント) / HTTPS (REST)
[FastAPI サーバー]  /opt/interview
    |-- セッション管理・モード制御・ターン管理
    |-- Live APIブリッジ (ブラウザ⇔Gemini Live のプロキシ)
    |-- 審判サービス (テキストLLM, マルチプロバイダ)
    |-- ペルソナ生成サービス
    |-- 即時チェッカー (禁止表現 regex)
    |-- SQLite (sessions.db)
[外部API]  Gemini Live / Gemini / OpenAI / Anthropic / xAI
```

UIフレームワークはStreamlitではなく**FastAPI＋素のブラウザフロントエンド**を採用する。理由: リアルタイム双方向音声（WebSocketストリーミング、バージイン）はStreamlitの実行モデルと相性が悪く、音声が主体機能である本アプリでは制約が大きいため。

### 6.2 ディレクトリ構成

```
/opt/interview/
├── .env                    # 環境設定（600, git管理外）
├── .venv/                  # Python仮想環境
├── docs/                   # 本仕様書・Cursor指示書
│   ├── SPECIFICATION.md
│   └── CURSOR_INSTRUCTIONS.md
├── .cursor/rules/project.mdc
├── app/
│   ├── main.py             # FastAPI エントリポイント
│   ├── config.py           # .env読込・役割別プロバイダ解決
```

## 10. 実装時判断事項（低リスクのため文書内確認とした項目）

| # | 事項 | 本仕様の判断 |
|---|---|---|

## 11. 制約・注意事項

- 面談内容は外部AI API（Google等）に送信される。**実在の人事評価情報・個人の機密情報を入力しない**運用を利用者に周知すること（S-01で同意取得）。
- 本アプリの講評は研修目的の参考情報であり、実際の人事評価・処遇判断には使用しない。
- 判定ルーブリックは中星工業マニュアル（2019.08版/2026.04改定資料）に準拠する。マニュアル改定時は `data/rubrics.json` を更新すること。
