## 0. このドキュメントの使い方

1. 「1. 環境設定(要記入)」のプレースホルダー(`【 】`で囲まれた箇所)をご自身の環境に合わせて書き換えてください。
2. 書き換え後、**「2. アプリケーション概要」以降を全てそのままCursorのチャット/Composerに貼り付けて**ください。
3. 1回で全機能を作らせず、まず「2〜7(基盤+取り込み)」→「8〜9(UI+生成ロジック)」→「10〜11(デザイン+QA)」→「12(コストシミュレーション)」→「13(堅牢性: タイムアウト・キャンセル)」の5段階に分けて投入すると、Cursor側の手戻りが少なくなります(一度に全部貼ってもCursorは対応できますが、レビューが大変になります)。

## 1. 環境設定(要記入)

```
TARGET_OS        = 【 Linux または Windows をここに記入 】
INSTALL_ROOT     = 【 Linuxの場合: /opt/任意のフォルダ名 ・ Windowsの場合: C:\ai-app\任意のフォルダ名 】
APP_FOLDER_NAME  = 【 任意のアプリフォルダ名(例: multi-llm-slide-gen) 】
LAN_PORT         = 【 Streamlit公開ポート番号(例: 8505) 】
LAN_BIND         = 【 社内LAN公開する場合は 0.0.0.0 / ローカルのみなら localhost 】
```

> 上記が確定すると、以下が自動的に決まります。
> - **Linux** の場合: `PYTHON_CMD=python3` / `VENV_ACTIVATE=source venv/bin/activate` / セットアップスクリプトは `setup.sh`
> - **Windows** の場合: `PYTHON_CMD=python` / `VENV_ACTIVATE=venv\Scripts\activate` / セットアップスクリプトは `setup.bat` と `setup.ps1` の両方

Cursorへの指示: **TARGET_OSの値に応じて、パス区切り文字・改行コード・セットアップスクリプトの構文を自動的に切り替えること。** 両OS分のコードを混在させず、TARGET_OSで指定された側のみを生成すること。

## 2. アプリケーション概要

`{INSTALL_ROOT}/{APP_FOLDER_NAME}` 以下に、以下の要件を満たすStreamlitベースのPythonアプリケーションを構築してください。

**目的**: 複数の異種ソースファイル(文書・表計算・音声・動画など)をアップロードし、生成目的・想定利用シーンを入力すると、Gemini / OpenAI / Anthropic(Claude) / xAI(Grok) の中から各処理工程に最適なLLMを自動選択(または手動指定)して、編集可能なPowerPoint(.pptx)資料を自動生成・ダウンロードできるアプリ。

**設計思想**: 単一のLLMに全工程を任せるのではなく、「文書解析」「内容統合」「構成設計」「日本語表現生成」「デザイン適用」「品質検証」という工程ごとに、利用可能なAPIキーの中から最も適したモデルを動的に割り当てる**オーケストレーション方式**とする。

## 6. オーケストレーション設計

### 6.1 工程定義(`config/task_routing.json`)

各処理工程に対して、**優先順位付きモデルリスト**(必ずしも最高性能順ではなく、用途適性順)をデフォルト値として定義する。実行時に「利用可能なAPIキー」でフィルタリングし、リストの先頭から使用可能な最初のモデルを採用する。

```json
{
  "file_parsing_text":      { "priority": ["gemini", "claude", "gpt4o"], "reason": "長文脈・多形式の一次解析に強い" },
  "audio_video_understanding": { "priority": ["gemini"], "reason": "ネイティブ音声/動画理解はGeminiのみ対応" },
  "content_synthesis":      { "priority": ["gemini", "claude"], "reason": "複数ソース横断の長文脈統合" },
  "slide_structure_planning": { "priority": ["claude", "gpt4o", "gemini"], "reason": "枚数・時間制約を踏まえた構成設計の精度" },
  "structured_json_payload": { "priority": ["gpt4o", "claude", "gemini"], "reason": "厳密なJSONスキーマ準拠率" },
  "japanese_narrative":     { "priority": ["claude", "gemini", "gpt4o"], "reason": "自然な日本語表現の質" },
  "design_visual_qa":       { "priority": ["claude", "gpt4o"], "reason": "画像入力によるレイアウト検証(Vision対応必須)" },
  "fallback":                { "priority": ["grok"], "reason": "他モデル全てキー未設定/エラー時の最終手段" }
}
```

このJSONはUIの「詳細設定」画面から編集可能にすること(後述9章)。

### 6.2 ルーティングロジック(`core/orchestrator.py`)

```
入力: task_name, 利用可能モデル一覧(.envから判定), ユーザーによる手動指定(任意)
処理:
  1. ユーザーがそのtask_nameに対して手動指定をしている場合 → そのモデルを使用(キーが無ければエラー表示)
  2. 手動指定が無い場合 → task_routing.json の priority リストを上から確認し、
     利用可能モデル一覧に存在する最初のモデルを採用
  3. 採用モデルでのAPI呼び出しが失敗した場合 → priority内の次の候補に自動フォールバック
  4. 全候補が失敗した場合 → fallback.priority(grok等)を試行、それも失敗ならUIにエラー表示し
     当該工程をスキップ可能とする(処理全体は止めない)
出力: 採用モデル名, 採用理由, 処理結果 をlogs/に記録し、UI上にも表示
```

### 6.3 手動オーバーライド(自動判定+手動切替)

UIのサイドバーまたは「詳細設定」エクスパンダーに、工程ごとのプルダウン(「自動」「Gemini」「Claude」「GPT-4o」「Grok」)を配置。「自動」選択時は6.2のロジックに従う。手動選択時はそのモデル固定。**処理実行中は、現在どの工程をどのモデルが処理しているかをリアルタイムでUI上に表示すること**(例: `🔄 内容統合中... (使用モデル: Gemini 2.5 Pro / 理由: 長文脈処理)`)。

## 7. ファイル取り込み仕様

### 7.1 対応形式と処理方法

| 形式 | 処理方法 |
|------|---------|
| .txt, .md | そのまま読み込み |
| .csv | pandasで読み込み、構造を保持したままテキスト化 |
| .pdf | pypdfでテキスト抽出。抽出文字数が極端に少ない場合(スキャンPDFの可能性)は警告を表示し、画像化+Vision対応モデルでのOCR処理にフォールバック |
| .docx | python-docxで段落・表・見出し構造を保持して抽出 |
| .pptx | python-pptxでスライドごとのテキスト・ノートを抽出 |
| .xlsx, .xlsm | openpyxlでシート・セル構造を保持して抽出(マクロ自体は実行しない、値のみ) |
| .mp3, .m4a | Gemini APIへ直接ファイルを渡し、ネイティブ音声理解機能で文字起こし・要約 |
| .mp4, .m4v | Gemini APIへ直接ファイルを渡し、ネイティブ動画理解機能で内容理解(音声+視覚情報) |

### 7.2 複数ファイル・異種形式の統合

アップロードされた全ファイルを `file_ingest.py` で個別にテキスト化した後、ファイル名・形式・抽出内容を構造化したリストとして `content_synthesizer.py` に渡し、6.1の `content_synthesis` 工程で1つの統合コンテキストとして要約・整理する。**ソースごとの矛盾点や情報の濃淡があれば、統合時にその旨を内部的に記録し、後続の構成設計工程で考慮できるようにすること。**

### 7.3 URL/リンクソース(物理ファイルとの併用可)

物理的なファイルアップロードに加え、以下2種類のURL入力をソースとして併用できるようにすること。`core/web_source_ingest.py` として新規モジュールを実装する。

**(a) YouTube動画の共有リンク**

- Gemini APIは2026年時点でYouTube URLを `file_uri` として直接渡す機能を提供している(プレビュー機能・無償提供だが将来的に料金・制限が変更される可能性がある旨をコード内コメントとUIの注記に明記すること)
- 対象動画は**公開動画のみ**(限定公開・非公開動画は読み込めない場合がある)。リクエストあたり扱えるYouTube動画は基本的に1本のため、複数のYouTube URLが入力された場合は**1本ずつ順番にAPI呼び出しを行い、結果を統合する**実装とすること
- 処理は6.1の `audio_video_understanding` 工程(Gemini固定)にそのままルーティングする
- 実装例(Python, `google-genai` SDK):
  ```python
  response = client.models.generate_content(
      model="gemini-3-flash-preview",  # config/design_system.py等で実際の利用可能モデル名を管理
      contents=[
          {"file_data": {"file_uri": youtube_url}},
          {"text": "この動画の内容を、後続のスライド資料作成に使えるよう詳細に要約してください。"}
      ]
  )
  ```
- エラーハンドリング: 非公開動画・著作権制限・年齢制限などでAPIがエラーを返した場合は、当該URLをスキップし「◯件のYouTube動画は読み込めませんでした」とUI上に警告表示。処理全体は止めないこと

**(b) 参考WebサイトのURL**

- YouTube以外の任意のWebページURLは、Gemini/Claude等のLLMに直接渡すのではなく、**アプリ側でHTML取得→本文抽出を行ってからテキストソースとして扱う**こと(LLM側のネイティブURL取得機能に依存すると、サイトによって取得可否や精度にばらつきが出るため)
- 取得・抽出には `requests`(または `httpx`)+ `trafilatura`(本文抽出ライブラリ)の組み合わせを推奨。ナビゲーション・広告・フッターなどのボイラープレートを除去し、本文のみを抽出すること
- 抽出後は7.1の通常テキストソースと同様に `content_synthesis` 工程に渡す(`file_parsing_text` 工程経由)
- エラーハンドリング: アクセス不可・ロボット排除設定(robots.txt)によるブロック・ペイウォール・タイムアウトなどが発生した場合は、当該URLをスキップしUI上に警告表示。処理全体は止めないこと
- **社内ネットワークからの外部URL取得を考慮**: 中星工業様のネットワーク環境(特にZGX Nano等の社内サーバーで稼働させる場合)は、ファイアウォール・プロキシ経由でしか外部HTTPS通信ができない可能性がある。`.env` に以下を追加し、未設定ならプロキシなしで直接通信を試みる設計とすること:
  ```
  HTTP_PROXY=
  HTTPS_PROXY=
  ```

### 7.4 UI側の入力欄追加(8.1への追記)

8.1のメイン画面「1. ファイルアップロード欄」の直下に、以下を追加すること:

- **YouTube動画URL欄**(複数行テキストエリア、改行区切りで複数URL入力可)
- **参考WebサイトURL欄**(複数行テキストエリア、改行区切りで複数URL入力可)

物理ファイル・YouTube URL・WebサイトURLは**いずれか1種類のみでも、自由に組み合わせて併用しても**処理を実行できること。全てのソースが空欄の場合のみ「生成開始」ボタンを無効化し、エラーメッセージを表示する。

### 7.5 複数ソース処理時の進捗表示と所要時間の目安

- **YouTube動画**: 1リクエスト1本の制約上、複数本(例: 5本)は逐次処理となる。進捗表示は6.3の工程別表示に準じ、「YouTube動画処理中 (3/5)」のように現在処理中の本数を明示すること
- **WebサイトURL**: HTML取得・本文抽出はAPI呼び出しを伴わないため、**並列処理(同時実行数3〜5程度を上限)**で高速化してよい。サイトへの過度な負荷を避けるため、同時実行数は `config/design_system.py` 等の定数として上限を設定すること
- 物理ファイル・YouTube動画・WebサイトURLを多数同時に併用する場合、全体の処理時間が数分〜十数分に及ぶ可能性があるため、UI上に「処理にはソース数に応じて数分程度かかる場合があります」という事前案内を表示し、処理中はページを閉じないよう注記すること

## 8. UI仕様(Streamlit画面構成)

### 8.1 メイン画面

1. **ファイルアップロード欄**(複数選択可、対応形式は7.1の一覧を明示)
2. **生成目的・用途の入力欄**(テキストエリア): 例「経営層向け、生成AI活用方針の説明」
3. **想定利用相手の入力欄**(テキストエリア): 例「役員5名、ITリテラシーは高くない」
4. **スライド枚数・時間設定**(9章で詳述)
5. **詳細設定(エクスパンダーで折りたたみ)**:
   - 工程別LLM手動オーバーライド(6.3)
   - デザインパレットの選択(10章のパレット一覧からの選択、または「内容に応じて自動選定」)
6. **「コストを見積もる」ボタン**: 12章のロジックで予算シナリオ一覧を表示し、ユーザーがシナリオ選択または予算上限を直接入力できるようにする(この段階ではLLM API呼び出しは発生しない。12.5参照)
7. **「生成開始」ボタン**(コスト見積もり確認後に活性化)
8. **進捗表示エリア**: 工程名・使用モデル・ステータス・現在の実コスト累計をリアルタイム表示
9. **「キャンセル」ボタン**(生成処理中は常時表示。13.3参照。押下時はコスト精算を待たず即座に処理を中断する)
10. **生成完了後**: プレビュー(スライドサムネイル画像)+ `.pptx` ダウンロードボタン + 最終確定コスト

### 8.2 セッション管理

生成処理は時間がかかるため、Streamlitの `st.session_state` で処理状態を保持し、ページ再読み込みや他操作で処理が中断されないようにすること。長時間処理(音声/動画ファイルが大きい場合等)は `st.spinner` ではなく工程別の進捗バー(`st.progress`)を使用すること。**実処理は13.2の通りバックグラウンドスレッド/プロセスで実行し、UIは定期ポーリングで状態を反映する設計とすること(UIスレッドのブロッキングによる「処理中」表示の固着を防ぐため)。**

## 9. スライド生成ロジック(枚数・時間の扱い)

### 9.1 入力項目

- **最低枚数**(数値入力、デフォルト未設定可)
- **最大枚数**(数値入力、デフォルト未設定可)
- **想定プレゼン時間**(分単位の数値入力)
- **「おすすめ構成で生成」チェックボックス**

### 9.2 「おすすめ構成で生成」モード

チェックボックスがONの場合、上記3項目の手動入力欄は無効化(グレーアウト)し、代わりに以下を自動算出する。**この算出はデフォルトで無料(LLM API呼び出しを行わないローカル処理)とすること。**

1. **デフォルト(無料)**: アップロードされた全ソースについて、文字数・見出し/段落数・(音声/動画は再生時間)といった**ローカルで取得可能な統計情報のみ**を用いて、簡易的な情報量(トピック数の目安)を推定する。LLMへの問い合わせは行わない
2. 1スライドあたり平均1〜1.5分の説明時間という業務向けプレゼンの一般的な目安を基準に、**推奨スライド枚数**と**推奨プレゼン時間**を算出
3. 算出根拠(「ソースの分量から◯枚構成・約◯分を推奨(簡易推定)」)をUI上に表示してからスライド生成を実行
4. **「AIで精密分析する」ボタン(任意・課金あり)**: より精度の高い推奨枚数が欲しい場合のみ、ユーザーが明示的にクリックすると `slide_structure_planning` 工程のLLMにソース内容を渡して再分析する。この場合は事前に「この分析には約$◯.◯◯の費用が発生します」とコストを明示し、ユーザーの確認後にのみ実行する
5. 生成完了後も、実際の構成枚数と推定時間を結果画面に明示すること

### 9.3 手動指定モードとの整合

手動で最低/最大枚数・時間を指定した場合は、`slide_structure_planning` 工程がその制約内に収まるよう構成を調整する。ソース情報量に対して枚数が著しく不足/過剰な場合は、生成前に警告(例: 「指定された枚数では情報を圧縮しすぎる可能性があります」)を表示すること。

## 10. デザインシステム仕様(品質担保の核心部分)

**この章はスライドのデザイン品質を左右する最重要部分です。`pptx_renderer.py` および `templates/layouts.py` にハードコード、または `config/design_system.py` の定数として実装し、LLMの自由出力に任せず必ず適用すること。**

### 10.1 配色ルール

- 1色が60〜70%の視覚的支配力を持ち、1〜2色の補助色、1色の鋭いアクセントカラーという比率を必ず守ること(全色を均等に使わない)
- パレットは「内容に応じて自動選定」モードの場合、以下の候補から生成目的・トーンに合わせて `content_synthesis` 工程の結果を踏まえてLLMに1つ選定させる:

| テーマ名 | Primary | Secondary | Accent |
|---------|---------|-----------|--------|
| Midnight Executive | `1E2761` | `CADCFC` | `FFFFFF` |
| Forest & Moss | `2C5F2D` | `97BC62` | `F5F5F5` |
| Coral Energy | `F96167` | `F9E795` | `2F3C7E` |
| Ocean Gradient | `065A82` | `1C7293` | `21295C` |
| Charcoal Minimal | `36454F` | `F2F2F2` | `212121` |
| Teal Trust | `028090` | `00A896` | `02C39A` |
| Cherry Bold | `990011` | `FCF6F5` | `2F3C7E` |

- タイトル・結論スライドは暗背景、本文スライドは明背景とする「サンドイッチ構造」、または全編暗背景で統一する「プレミアム構造」のいずれかを選択し、混在させないこと

### 10.2 タイポグラフィ

| 要素 | サイズ |
|------|--------|
| スライドタイトル | 36〜44pt, bold |
| セクション見出し | 20〜24pt, bold |
| 本文 | 14〜16pt |
| キャプション | 10〜12pt, 控えめな色 |

**フォントは以下の安全リストを優先使用**(PowerPoint実機とQA用LibreOffice双方で幅が一致するため、文字あふれのリスクが低い): Arial, Calibri, Cambria, Times New Roman, Courier New, Bookman Old Style, Century Schoolbook。これ以外のフォントを使う場合はテキストボックスに約10%の余裕を持たせること。

### 10.3 レイアウト・余白

- 最小余白0.5インチ、コンテンツブロック間0.3〜0.5インチ(混在させず統一)
- 本文は左揃え(中央揃えはタイトルのみ)
- 同一レイアウトの連続使用を避け、2カラム/アイコン+テキスト行/グリッド/統計強調などをスライドごとに変化させること
- **全スライドに最低1つの視覚要素**(図形、アイコン、チャート、画像のいずれか)を含めること。文字だけのスライドは禁止

### 10.4 禁止事項(明示的に避けること)

- タイトル下のアクセントライン(AI生成スライドの典型的特徴のため使用しない)
- 装飾的なカラーバー・サイドストライプ・縁取りのみのボーダー
- ベージュ/クリーム系の背景をデフォルトにしない(無指定時は白 `FFFFFF` または指定パレットを使用)
- テキストがボックスからはみ出した状態での出力(はみ出す場合はフォントサイズ縮小・分割・ボックス拡大のいずれかで必ず解消)

### 10.5 レイアウト生成方式: 自由生成モード(デフォルト)+ テンプレートへの自動フォールバック

スライドごとのレイアウトは、**LLMにその場でpython-pptxコードを生成させる「自由生成モード」をデフォルト**とする。これは固定テンプレートよりも表現の幅を広げ、依頼内容に応じた独自性の高いレイアウトを実現するための選択。ただし無人実行であることを踏まえ、以下のセーフティネットを必ず実装すること。

**処理フロー(`pptx_renderer.py` + `templates/layouts.py`)**:

1. `slide_layout_code_generation` という工程を `config/task_routing.json` に新規追加し、優先順位を `["claude", "gpt4o", "gemini"]` とする(コード生成・デザイン判断の精度を踏まえた順序。手動オーバーライドも6.3と同様に可能)
2. 選定されたLLMに対し、当該スライドの内容・10.1〜10.3のデザインシステム定数(配色・タイポグラフィ・余白ルール)・10.4の禁止事項を渡し、そのスライド1枚分を構築するpython-pptxコード(関数本体)を生成させる
3. 生成コードは**サンドボックス化したサブプロセスで実行**し、ファイルI/Oやネットワークアクセスなど、スライド構築に不要な操作を行わないことを確認した上で実行すること(セキュリティ上の最低限の配慮として、python-pptxのAPI呼び出しのみに限定する簡易バリデーションを行う)
4. 実行時に例外が発生した場合、エラーメッセージをLLMにフィードバックして**再生成を最大3回まで試行**する
5. 3回失敗した場合は、`templates/layouts.py` に用意した定型レイアウト関数(最低8〜10パターン用意すること: 2カラム、アイコン+テキスト行、2x2/2x3グリッド、統計強調、比較カラム、タイムライン等)の中から内容に最も近いものへ**自動フォールバック**し、処理を止めない
6. どちらの方式で生成されたスライドかを `logs/` に記録すること(自由生成成功/フォールバック発生の比率を後から把握できるようにするため)

この方式により、通常は自由生成によるWeb版に近い表現力を確保しつつ、生成コードの不具合でアプリ全体が停止することは防げる。

## 11. QA(品質検証)ループ

`qa_checker.py` に以下の検証フローを実装すること:

1. 生成した `.pptx` をLibreOffice(`soffice --headless --convert-to pdf`)でPDF変換
2. Popplerの `pdftoppm` で各スライドをJPEG画像化
3. 6.1の `design_visual_qa` 工程で選定されたVision対応モデルに画像を渡し、以下を検証させる:
   - 要素の重なり、テキストのはみ出し・切れ
   - 余白不足(0.3インチ未満の近接)
   - コントラスト不足(背景と文字色)
   - レイアウトの不整合
4. **検証は最大5回まで繰り返す**。UI上には常に現在の検証回数を `(現在の回数/5)` の形式で表示すること(例: `QAチェック中... (2/5)`)
5. **早期終了**: いずれかの回で「ユーザー視点で問題のある欠陥が検出されない」と判定された時点で、5回に満たなくてもその回で検証を終了し、結果を確定する(例: 2回目で問題なしと判定されれば `(2/5)` 表示のまま完了)
6. 各回で検出された問題のうち、該当スライドのみ修正してから次の回の検証に進む。**前回までに「問題なし」と判定済みの観点は再検証対象から除外し、新たに生じた箇所のみ確認する**ことで、回数を重ねても無駄な手戻りを防ぐ
7. 5回検証してもなお解消しない問題が残る場合は、その時点のファイルを確定版として確定し、UI上に「QAチェック: 5回実施、未解消の軽微な問題が◯件あります」のように残存問題を明示した上でダウンロード可能にする(処理自体は止めない)
8. 検証結果と修正履歴は全回分を `logs/` に記録し、完了後にUI上で「QAチェック: ◯回実施、◯件の問題を検出・修正済み」のように要約表示

## 12. コストシミュレーション機能

「生成開始」前に、想定コストを事前見積もりし、複数の予算シナリオをユーザーが選べるようにする。あわせて、見積もりはあくまで予測であるため、**実行中の実コスト監視による安全装置**もセットで実装する。

### 12.1 料金テーブル(`config/pricing_rates.json`)

各プロバイダ・モデルごとの単価(入力/出力トークン単価、画像1枚あたり単価、音声/動画は秒あたり単価)を定義する。**LLMの料金は頻繁に改定されるため、ハードコードせず必ず外部設定ファイルとして分離し、UIの「詳細設定」から確認・編集できるようにすること。** ファイル冒頭のコメントに「料金は変更される可能性があるため、各社の公式料金ページで定期的に(例: 月1回)確認・更新してください」という注記を必ず入れる。

```json
{
  "gemini": { "input_per_1m": 0.0, "output_per_1m": 0.0, "video_per_sec": 0.0, "audio_per_sec": 0.0 },
  "claude": { "input_per_1m": 0.0, "output_per_1m": 0.0, "image_per_unit": 0.0 },
  "gpt4o":  { "input_per_1m": 0.0, "output_per_1m": 0.0, "image_per_unit": 0.0 },
  "grok":   { "input_per_1m": 0.0, "output_per_1m": 0.0 }
}
```

(数値は導入時に各社公式ページの最新単価を確認の上、鈴木さんご自身で記入してください。Cursorには「0.0をプレースホルダーとして残し、初回起動時にUI上で『料金が未設定です』と警告表示する」処理を実装させること。)

### 12.2 事前見積もりロジック(`core/cost_estimator.py`)

**この見積もり処理自体はLLM APIを呼び出さず、無料で実行できる設計とすること。** 「生成開始」ボタンの前に「コストを見積もる」ボタンを設け、以下を計算する:

1. **入力トークン**: アップロード済み全ソース(物理ファイル・YouTube・WebサイトURL)を実際にLLMへ渡す前に、テキスト換算量からトークン数を概算(英数字は1トークン≒4文字、日本語は1トークン≒1〜1.5文字程度を目安とした概算関数を実装。正確な計算はプロバイダのトークナイザーに依存するため、あくまで概算である旨をUI上に明記する)。これはローカルの文字数カウントのみで完結し、API呼び出しは発生しない
2. **音声/動画**: ファイルの再生時間は `ffprobe`(またはPythonの`mutagen`/`moviepy`等)で**ローカルにメタデータのみ取得**し、Gemini APIの目安単価(動画は1秒あたり約300トークン相当、音声は1秒あたり約32トークン相当という公開資料の目安を `config/pricing_rates.json` 内の換算定数として保持)に再生時間を掛けて推定する。**実際にGemini APIへファイルを送信して内容分析させることはこの見積もり段階では行わない**
3. **YouTube動画**: 動画の長さは、YouTubeのoEmbed API(APIキー不要・無料)等で**メタデータのみ取得**し、2.と同様の換算で推定する。Gemini APIによる実際の動画理解はこの段階では呼び出さない
4. **WebサイトURL**: 本文取得自体はHTTPリクエスト(無料)で完結するため、見積もり段階で先に本文を取得して文字数カウントしてよい(取得結果はキャッシュし、実際の生成時に再取得しない設計にすることで二重アクセスを避ける)
5. **出力トークン**: 想定スライド枚数(9章で決定した枚数、または「おすすめ構成」のローカル推定枚数)を基準に、工程ごとの平均出力トークン数(スライド1枚あたりのナレーション・構造化JSON・レイアウトコードの目安値を `config/pricing_rates.json` または別定数ファイルに保持し、運用しながら実測値で補正できるようにする)を掛け合わせて推定
6. **QA工程**: 画像入力1枚あたりの目安コスト × スライド枚数 × 想定QA回数(デフォルトは11章の上限5回として最大値を見積もる)
7. 各工程で実際に採用される予定のモデル(6.1のtask_routingに基づく)の単価を当てはめ、工程ごとの見積もり額を合算して**総額(基準シナリオ)**を算出する

**例外**: 9.2の「AIで精密分析する」ボタンを使った場合のみ、その分析コストが見積もり段階で発生する。この場合は実行前に必ず個別に金額を提示し、ユーザーの同意を得てから実行すること。

### 12.3 予算シナリオの表示

見積もり後、以下のように**複数のレバー(調整項目)を個別にON/OFFした場合の概算額**を一覧表示し、ユーザーが選択できるようにする。レバーの種類・表示順は固定2つに限定せず、設定ファイルで増減可能な可変リストとして実装すること。

| シナリオ例 | 内容 | 概算額 |
|-----------|------|--------|
| 費用度外視 | 現行設定のまま(自由生成モード+QA最大5回+指定/推奨枚数) | $◯.◯◯ |
| 枚数を抑える | スライド枚数を範囲の下限値まで圧縮 | $◯.◯◯ |
| デザイン確認を1回に制限 | QA検証を最大5回→1回に制限(11章の上限値を一時的に1へ変更) | $◯.◯◯ |
| レイアウトをテンプレート固定 | 10.5の自由生成モードをオフにし、テンプレート関数のみ使用 | $◯.◯◯ |
| 最安構成 | 上記の制限を全て組み合わせ | $◯.◯◯ |

**任意の予算上限を直接入力する方式**も併設すること。例えば「$2」と入力すると、レバーの**影響度が小さい順**(デフォルト優先順位: ①QA回数を減らす → ②枚数を圧縮する → ③レイアウトをテンプレート固定にする)に自動で制限をかけ、予算内に収まる構成を提案する。この優先順位はUIの詳細設定から変更可能にすること(鈴木さんが「デザイン品質より枚数を優先したい」等、状況に応じて調整できるようにするため)。

ユーザーはシナリオを選択(またはカスタム構成を確定)した上で、初めて実際の生成処理を開始する。

### 12.4 実行中のリアルタイム監視(安全装置)

事前見積もりは予測であるため、実行中に以下を必ず実装すること:

1. 各LLM API呼び出しのレスポンスに含まれる実トークン使用量(usageフィールド等)を都度取得し、`config/pricing_rates.json` の単価で実コストを積算する
2. UI上に「現在の実コスト: $◯.◯◯ / 見積もり: $◯.◯◯」をリアルタイム表示する
3. 実コストが選択シナリオの見積額の**1.3倍**を超えた時点で処理を一時停止し、「想定より費用がかさんでいます。続行しますか?」とユーザーに確認を求める(自動停止ではなく確認を挟むことで、処理の継続可否をユーザーが判断できるようにする)
4. 最終的な実コストは生成完了後にUI上で確定額として表示し、`logs/` にも記録する

### 12.5 見積もり段階でのキャンセル

見積もり結果(シナリオ一覧)の表示画面には、必ず「キャンセル/最初からやり直す」ボタンを併設すること。この時点ではLLM API呼び出しが発生していない(12.2参照)ため、キャンセルしても課金は一切発生しない。ユーザーがシナリオを選ばずにキャンセルした場合、アップロード済みソース・入力済み設定はセッション内に保持したまま入力画面に戻り、再度条件を変えて見積もり直せるようにすること。

## 13. 処理の堅牢性(タイムアウト・スタック防止・ユーザーキャンセル)

### 13.1 タイムアウト設計

各LLM API呼び出し・各ファイル処理(音声/動画分析、Webサイト取得等)に対し、工程ごとの**上限時間**を `config/pricing_rates.json` と同様の設定ファイル(例: `config/timeouts.json`)で定義すること。目安:

| 工程 | タイムアウト目安 |
|------|-----------------|
| テキスト系LLM呼び出し(構成設計・JSON生成・ナレーション等) | 60〜120秒 |
| 音声理解(Gemini) | ファイル長に応じて可変(例: 再生時間の3〜5倍、上限10分) |
| 動画理解(Gemini・YouTube含む) | ファイル長に応じて可変(上限15分) |
| WebサイトURL取得 | 15〜30秒 |
| レイアウトコード生成・実行(10.5) | 生成30秒、実行5秒 |
| QA画像検証(11章) | 1回あたり60秒 |

上限時間を超えた場合は例外として扱い、**6.2のフォールバックロジック(次の候補モデルへ切替、または当該工程をスキップ)に必ず接続すること**。「処理中」のままタイムアウト処理に入らない実装は不可とする。

### 13.2 スタック検知・ウォッチドッグ

- パイプライン全体の実行は、Streamlitのメインスレッドをブロックしないよう**バックグラウンドスレッドまたは別プロセスで実行**し、UI側は定期的(例: 1〜2秒ごと)に進捗状態をポーリングして表示を更新する設計とすること(`st.spinner`だけに頼ると、内部処理が固まった場合にUIも反応しなくなる典型的な不具合パターンになるため)
- 全体処理に対しても**最終防衛ラインとしての全体タイムアウト**(例: 30分)を設定し、これを超えた場合は強制的に処理を中断、UI上に「処理が想定時間を超えたため中断しました」と表示し、それまでに完成している部分(生成済みスライド等)があればダウンロード可能にすること
- バックグラウンド処理が異常終了(プロセスクラッシュ等)した場合も、フロント側がそれを検知できるよう、定期ポーリングで一定時間応答がなければ「処理が異常終了した可能性があります」と表示し、再試行またはキャンセルを促すこと

### 13.3 ユーザーによるキャンセル

- 生成処理中は常時画面上に**「キャンセル」ボタン**を表示すること
- クリックされた時点で、実行中のバックグラウンド処理に対して中断シグナル(例: `threading.Event`または別プロセスの場合はプロセス終了)を送り、**現在実行中のAPI呼び出しが完了次第、それ以降の工程には進まず即座に処理を終了する**こと
- **それまでに発生した実コストの計算・表示・保存は行ってよいが、キャンセル自体の可否や処理速度に影響させないこと**(コスト精算を待ってからキャンセル完了とするような設計にしない)
- キャンセル後は一時ファイル(`temp_uploads/`等)をクリーンアップし、UIを初期状態に戻して再度新しい生成を開始できるようにすること

## 14. エラーハンドリング・ログ

- 各LLM呼び出しは必ずtry-exceptで囲み、失敗時は6.2のフォールバックロジックに従う
- ログは `logs/{YYYYMMDD_HHMMSS}.log` に、工程名・使用モデル・処理時間・成否を記録

## 15. セットアップスクリプト要件

TARGET_OSの指定に応じて、以下のいずれかのみを生成すること。

**Linuxの場合(`setup.sh`)**:
- venv作成・有効化・pipインストールを自動実行
- `.env` が無ければ `.env.example` からコピー
- `soffice` / `pdftoppm` の存在確認(無ければインストールコマンド例を表示するのみ)
- 実行権限付与の案内(`chmod +x setup.sh`)も併記

**Windowsの場合(`setup.bat` および `setup.ps1`)**:
- 同等の処理をバッチファイルとPowerShellスクリプト両方で用意(社内ポリシーでPowerShell実行が制限されている場合への配慮)

## 16. 実装時の注意事項(Cursorへの指示)

- 段階的に実装すること。まず3章(ディレクトリ構成)・5章(.env)・6章(オーケストレーション基盤)を動く最小構成として作り、その後7章(ファイル取り込み)、8〜9章(UI・生成ロジック)、10〜11章(デザイン・QA)、12章(コストシミュレーション)、13章(堅牢性)の順に拡張すること
- 各LLM APIクライアントは `llm_clients.py` に統一インターフェース(例: `generate(task_name, prompt, files=None) -> str`)でラップし、`orchestrator.py` 以外のモジュールが個別のSDKを直接呼ばないようにすること(将来的なモデル追加・差し替えを容易にするため)
- xAI(Grok)はOpenAI互換エンドポイントのため、`openai` SDKの `base_url` を `https://api.x.ai/v1` に切り替える形で実装すること
- コード中のコメントは日本語で記述すること
- 10章のデザインルールは「LLMへの指示」ではなく**コード側の制約・テンプレート関数**として強制し、LLMの出力ゆらぎに左右されない設計にすること
- 10.5の自由生成モードは、生成コードのサンドボックス実行(try-except + タイムアウト + API呼び出し範囲の簡易バリデーション)を**後回しにせず最初から実装**すること。ここを疎かにすると、不正なコード実行によるアプリ全体のクラッシュやハングにつながる
- 12章のコスト見積もりは**目安であり保証ではない**ことをUI上に明記すること。実コストとの乖離が生じうる前提で12.4のリアルタイム監視を必ずセットで実装し、見積もりのみで安全と判断しないこと
- 13章のバックグラウンド実行・タイムアウト・キャンセル機構は、機能の「おまけ」ではなく**8.2のセッション管理と一体で最初から設計**すること。後付けで非同期化しようとすると大規模な作り直しが必要になりやすい
